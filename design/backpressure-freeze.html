Freeze-on-overload, via the unified intent model

The decision that reshaped this feature: every player input becomes a fire-and-forget intent — recorded as per-actor state, resolved only in the tick. Once that's true, the overload-freeze stops being a special mechanism and becomes a one-liner: when the Datastore is Backpressured, skip the tick. This file maps today's runtime, the model change, and how the freeze falls out of it.

1. The shift: from two input models to one

InputTodayIntent model

harvest/build/damage resolve synchronously on receipt: mutate ECS, emit PersistEvents, reply with Ok/reason enqueue onto the actor's bounded FIFO; :noreply; resolved in the tick changed
success result already async (SelfInventory + chunk deltas) identical — async same

freeze handling — (nothing reads mode()) tick skipped while Backpressured the feature
per-verb frozen() gates — not needed — nothing resolves outside the tick dropped

Inbound wire shape is unchanged (same four events). What changes: the server route() enqueues instead of resolving; the synchronous reply schemas for the three verbs leave the contract; one ActionRejected event joins it. Player input now mirrors the NPC path — drive_npcs already writes intent before the integrator (world.rs:681); the glossary's "an NPC is simulated exactly like a Player" finally holds for actions too.

3. Why the freeze self-relieves

Flowing
full tick; actions resolve, writes flow
→
pending ≥ 1000
Backpressured
tick body skipped → no new writes; flush keeps draining
→
pending < 200
(flush drains it)
Flowing
tick resumes; queued intents drain, intact

A frozen world resolves no actions and integrates no movement, so it emits zero new PersistEvents. The unconditional flush on the freeze path drains the backlog under n_low and the Mode flips back on its own — no timer, no external nudge. Intents that arrive during the freeze sit in their queues (up to cap) and resolve the moment the tick resumes; surplus gets queue_full. Standing Velocity survives, so players continue in their last direction on resume.

4. The tick, two modes

Flowing tick

  • clock_ms += 50, tick_count++
  • drive_npcs → resolve action queues → integrate movement
  • portals, wildlife, overlay
  • drain_persistence → maybe_flush

Frozen tick

  • clock & tick_count untouched
  • no NPCs, no action resolution, no movement
  • datastore.flush() — unconditional
  • if now Flowing → next tick is a normal Flowing tick, intact

5. Proving test (re-spec)

players_freeze_under_overload_and_resume_intact — sim/tests/stories.rs:245

  1. Sim with low with_thresholds so overload is deterministic; connect a few players sharing the world authority.
  2. Drive pending past n_high → assert mode()==Backpressured.
  3. Enqueue move + harvest/build intents; tick repeatedly. Assert: positions do not advance, clock_ms constant, no PersistEvents produced, no inventory/structure changes.
  4. Assert no state dropped/corrupted: pre-freeze pending + durable intact; queued intents still pending (≤ cap), surplus saw queue_full.
  5. Let the unconditional flush drain below n_low → assert resume: clock advances, queued actions resolve in order, movement continues from standing Velocity; pre-freeze actions intact.

Drives Sim directly — confirming the freeze lives in the tick (the authority), not the wire layer.

6. Scope & remaining work

  • This is bigger than the original PLAN bullet. "Wire the existing Mode to a freeze" became "migrate player input to the intent model, whereupon the freeze is skip-the-tick." Cleaner and more determinism-by-construction, but the real work is the input-model migration, not the freeze.
  • Wire contract churn: remove synchronous reply schemas for harvest/build/damage; add the ActionRejected event; regenerate contract.json + update the client (stop awaiting verb replies, handle the new event).
  • Out of v1 (noted): per-cluster backpressure (only the overloaded Island freezes) — needs per-cluster write attribution; remains v2.
  • Client UX for ActionRejected reasons (queue_full = "world busy" vs gameplay reasons) — client-side, follow-up.
